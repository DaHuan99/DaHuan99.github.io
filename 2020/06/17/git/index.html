#git 命令

git add 文件名 添加到暂存区
git commit -m ‘描述’

git status 查看文件状态
git add -u （将已经进行 git 管理的文件，修改的全部提交）
git mv 文件名 新的文件名

git log –oneline 简洁的看 log 记录
git log –oneline -n2 分支最近两条简洁 log 记录
git log –oneline –all -n2 –graph 所有分支最近两条简洁 log 记录，带分支线
git branch -v 查看分支
git checkout -b 分支名称 log 号码
git branch -av 查看所有分支
git checkout 分支名 切换分支

.git 文件
1.cat HEAD 当前分支(上级目录直接 cat .git/HEAD)
1.cat config 当前用户公共信息(上级目录直接 cat .git/config)

git cat-file -p log 查看 commit 文件内容
git diff log1 log2 比较差异
git diff –cached 查看暂存区和 git 版本的修改差异
git diff 查看工作区和暂存区的差异
git diff –fileName 查看工作区和暂存区指定文件的差异
git diff 分支 1 分支 2 –fileName 查看两个分支的指定文件差异
git branch -d 删除分支
git branch -D 删除分支
git commit –amend 修改最近一次的 commit

git rebase 变基的使用
git rebase -i log
　　 pick：保留该 commit（缩写:p）
　　 reword：保留该 commit，但我需要修改该 commit 的注释（缩写:r）
　　 edit：保留该 commit, 但我要停下来修改该提交(不仅仅修改注释)（缩写:e）
　　 squash：将该 commit 和前一个 commit 合并（缩写:s）
　　 fixup：将该 commit 和前一个 commit 合并，但我不要保留该提交的注释信息（缩写:f）
　　 exec：执行 shell 命令（缩写:x）
　　 drop：我要丢弃该 commit（缩写:d）
git rebase 合并 commit
合并连续的 commit
git rebase -i log
设置 log s -> 第一行为什么要合并
合并间隔的 commit
git rebase -i log
将要合并的 log 移动到需要合并的第一条一起
git rebase –continue

git rebase –abort 执行之后，本地内容会回到提交之间的状态，也就是回到以前提交但没有 pull 是的状态，简单来说就是撤销 rebase。
git rebase –skip 引起冲突的 commits 会被丢弃，对于本文应用的例子来说开发者 A 对 c.sh 文件的 commit 无效，开发者 A 自己修改的部分全部无效，因此，在使用 skip 时请慎重。
git rebase –continue 执行完$git pull –rebase 之后，本地如果产生冲突，手动解决冲突之后 git rebase –continue

git 的文件放弃修改和删除
git reset Head 放弃暂存区
git reset Head –filename 放弃暂存区制定文件
git reset –hard commitId 放弃工作区和暂存区的 commit
git checkout —- fileName 取消对当前目录下工作区文件的修改
git reset –hard HEAD 重置工作区

git stash apply 释放最近一次的 stash 不会消除 stash list 数据
git stash pop 释放最近一次的 stash 消除 stash list 数据

git flow

git-flow是一套 git 使用的规范； Git Flow 建议，主要的分支有 master、develop、hotfix、release 以及 feature 这五种分支，各种分支负责不同的功能。其中 Master 以及 Develop 属于长期分支，因为他们会一直存活在整个项目开发中，而其它的分支大多会因任务结束而被删除。
###Master 分支
git 仓库初始化就存在的分支，主要是用来放稳定、随时可上线的版本。这个分支的来源只能从别的分支合併过来，开发者不会直接 Commit 到这个分支。因为是稳定版本，所以通常也会在这个分支上的 Commit 上打上版本号标籤。

###Develop 分支
这个分支主要是所有开发的基础分支，当要新增功能的时候，所有的 Feature 分支都是从这个分支切出去的。而 Feature 分支的功能完成后，也都会合併回来这个分支。

###Feature 分支
当要开始新增功能的时候，就是使用 Feature 分支的时候了。Feature 分支都是从 Develop 分支来的，完成之后会再併回 Develop 分支。

###Release 分支
当认为 Develop 分支够成熟了，就是到了发布一个新 Release 的时候，基于 Develop 分支创建一个 Release 分支，Release 分支在测试过程中可能会修改，完成 Release 后，合并到 Master 和 Develop 分支。

###Hotfix 分支
当在生产环境发现新的 Bug 时候，需要创建一个 Hotfix 分支, 完成 Hotfix 后，合并回 Master 和 Develop 分支，所以 Hotfix 的改动会进入下一个 Release。
